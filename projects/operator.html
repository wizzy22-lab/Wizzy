Overview

Project Summary

Transforming intuition-driven HVAC management into a predictable, decision-ready structure

This project started with a question from my teammate: "Why do people adjust their HVAC systems on intuition — without any clear criteria?" We considered who this problem should be solved for, and decided to target small business owners rather than residential users.

Running my own store, I kept adjusting the HVAC — but always based on intuition, with no clear criteria to guide me. That experience aligned directly with the problem. We concluded that HVAC isn't just about control — it's about a structure where making a judgment is inherently difficult.

In a context where too many variables make it hard for users to judge on their own, we shifted the approach: the system proposes the optimal choice first, and the user simply reviews and approves.

Problem Definition

Small business owners can control their HVAC systems — but without knowing the outcome of each choice, they cannot make a genuine judgment.

Because they can't see how each decision affects cost and comfort, users are unable to compare options — and end up repeating adjustments instead.

Ideate

Show What to Do Now — and What Will Happen

01

Solution Direction

Rather than showing HVAC data for users to interpret, the system makes the judgment first — and the user simply approves.

Building on the problem definition, we designed a structure that treats data not as raw information, but as a computed input — calculating the outcome of each option in terms of comfort and cost, then presenting a single optimal choice.

Core

The user doesn't make the judgment — the system does. The user only confirms.

02

Mind Mapping & Functional

HVAC decisions involve multiple factors simultaneously — outdoor environment, store conditions, cost — making it impossible to judge by any single criterion.

We mapped all the variables that influence HVAC operation and structured how each affects comfort and cost. From this, we designed a functional architecture that considers multiple variables at once and connects them into a single judgment.

03

Scenario & Prioritization

Based on real store operation scenarios, we defined only the features users truly needed as the MVP.

We analyzed the user journey to identify the key situations when HVAC adjustments occur: peak customer hours, sudden changes in outdoor temperature, and high electricity rate periods. Features were defined and prioritized based on the actions required in each scenario.

Core

Not every possible feature — only those directly needed for real decision-making were included in the MVP

04

Ideation & User Flow

Using Crazy 8, we explored main screen structures and decided on a flow that integrates judgment, choice, and outcome in a single view.

Comparing multiple layouts, we explored both a structure that split decision information across steps and one that displayed everything at once. Through internal team discussion, we initially determined that showing judgment, choice, and outcome simultaneously was better suited for faster decision-making. This was later validated in user testing, where the structure confirmed to genuinely accelerate the decision process.

05

IA & System Design

Rather than separating decision information from the action, we integrated both into a single continuous flow.

We reduced the steps between reviewing information, interpreting it, and making a selection — designing a structure that leads directly to a choice.

System Flow operator/ideate/system-flow.jpg
06

Wireframe & Design System

Status, outcomes, and choices were integrated into a single screen — enabling immediate judgment and selection.

The wireframe was structured so that current status (temperature and humidity), expected outcomes (comfort and cost), and available options (mode and settings) could all be viewed and acted on from a single screen.

The design system was built around information hierarchy and comparative structure — to support fast judgment without friction. To allow users to quickly distinguish decision-critical information, we clearly separated status, outcome, and choice — and applied a numeric + delta-based display format for intuitive comparison. Color and typography were used to distinguish current status, recommended choice, and expected outcome — so users could understand at a glance, without interpretation.

Iteration

Insight & Iteration

Testing Showed Where the Structure Still Got in the Way

The first version revealed two structural issues — both about flow. Places where small business owners had to break their thinking to navigate the interface.

Iteration 01

To keep the judgment flow intact, we reduced screen transitions and consolidated actions into a single context

In the original structure, completing a single decision required navigating across multiple screens — breaking the judgment flow in the process. To fix this, we consolidated related actions into a single screen so the decision process could flow without interruption.

Iteration 02

We moved the 'outcome' — what users look for first — to the top of the screen

In the initial layout, expected outcomes (comfort and cost) were placed at the bottom. Testing revealed that users immediately wanted to know 'what will actually happen' upon opening the app. Based on this, we moved outcome information to the top — redesigning the flow so users see the result first, then proceed to make a choice.

Reflection

01

How you define the problem determines every design decision that follows

We initially approached this as an HVAC control problem. Research revealed that the issue wasn't about operation — it was a structure where users couldn't make a judgment about what to do. Shifting that frame redirected us from designing features to designing a judgment flow. What to leave out became clearer than what to build.

02

Data only has meaning when it leads to a decision

Operators already had access to temperature, usage, and cost data. But none of it was connected to the outcome of each choice — making it impossible to use in actual decision-making. This project taught us that the problem wasn't a lack of data, but a structure that failed to link choices to results.

03

Users struggle more with deciding than with acting

Existing systems were designed to let users operate the controls. But in practice, the harder challenge wasn't how to adjust — it was deciding what to choose at all. Understanding this difference shifted our approach: instead of helping users judge, we designed a structure where the system judges first and the user approves.

04

Simplifying the judgment flow matters more than adding features

We could have added more features — but we deliberately removed anything that clouded the judgment flow. By reducing the steps between reviewing information, interpreting it, and selecting, and by integrating status, outcome, and choice into a single screen, we compressed the decision-making process. The result: users could make a choice with a single judgment instead of repeated adjustments.

05

Reducing scope isn't about giving up features — it's about making judgment clearer

In designing the MVP, we chose to include only the features directly needed for decision-making rather than everything possible. The features we cut made the remaining ones clearer in both role and flow — resulting in a structure where users could make faster, more confident decisions.
